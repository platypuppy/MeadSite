Grading Codes

What Do They Mean?

There are several aspects on which all of your programming assignments will be graded. Each aspect is associated with a code, usually 2 letters, but sometimes more. The meaning of the most common ones are explained below.
  • CW - Compiler warnings. This is no longer necessary since programs that won't compile cleanly will be rejected by the submission server.

  • NC - No Compile (error). This is no longer necessary since programs that won't compile will be rejected by the submission server.

  • OUT Output - Your output did not match the expected output. If you only had a very minor discrepency, you may lose as few as one point. If you had no output or all of it was incorrect, you may lose all of the points. A diff tool is always used to verify the output.

  • LEAK - There are memory leaks as reported by Valgrind or Dr. Memory

  • FHC File Header Comment - Each program file must include a file header that includes the information specified. This is an example of a file header from CS120. All lines are required: 
    /*****************************************************************************
filename    foo.cpp
author      Artie Fufkin
DP email    afufkin2@digipen.edu
course      CS170 
assignment  1
due date    9/19/2026

Brief Description:
  This program calculates the distance to the moon using only a compass
  and a straight-edge. Functions include:
  - Distance
  - Calculate
  - Convert
  - Convolve
  - Contort

*****************************************************************************/

    Doxygenized version: 

    /****************************************************************************/
/*!
\file    foo.cpp
\author  Artie Fufkin
\par     DP email: afufkin2@digipen.edu
\par     Course: CS170
\par     Section: A
\par     Assignment: 1
\date    9/19/2026

\brief
  This program calculates the distance to the moon using only a compass
  and a straight-edge. Functions include:
  
  - Distance
  - Calculate
  - Convert
  - Convolve
  - Contort
 
*/
/****************************************************************************/
  • FNH Function Header Comment - Each function must have a comment describing its purpose, return values (if any), and any input/output parameters. Here's an example from CS120. All lines are required: 
    /*****************************************************************************
   Function: EDetect
   
Description: Transforms a bitmap picture by darkening the edges that
             were detected by the edge-detect algorithm.

     Inputs: picture    - A pointer to the image that will be transformed.
             resolution - The degree to which the transformation will occur.

    Outputs: The number of processor cycles required by the algorithm.
*****************************************************************************/
    Doxygenized version: 
    /****************************************************************************/
/*!
  \brief   
    Transforms a bitmap picture by darkening the edges that were detected by 
    the edge-detect algorithm.

  \param picture
    A pointer to the image that will be transformed. 
      
  \param resolution
    The degree to which the transformation will occur.

  \return
    The number of processor cycles required by the algorithm.
*/
/****************************************************************************/

  • CRASH - The program crashed. This may or may not be easily detectable, depending on the nature of the crash.

  • HANG - The program hung. This is no longer necessary as the Watchdog timer terminates a hung or inefficient program. Points are deducted from the test that caused the hang.

  • ID Internal Documentation - Essentially, these are comments in the code. All code is required to have some documentation in the form of inline comments. Code that has no comments or very few will lose points.

  • CQ Code Quality - This is simply a gauge of the quality of your code. I posted a link to the Code Quality Guidelines which list some of these items. Not all will be applicable at this time. As we progress, more and more of these should concern you.

  • DW Doxygen Warnings - Doxygen may emit warnings just like the compiler does. You must not have any Doxygen warnings.

  • MEMBUGS Dr. Memory or Valgrind problems - These tools detect problems with your code at runtime. These can include things overwriting the end of an array or trying to free memory using free when the memory was allocated with new. These deductions are separate from memory leak deductions listed above.

  • EC - Extra Credit. Some assignments have extra credit tests.
